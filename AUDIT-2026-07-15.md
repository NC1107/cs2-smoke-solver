# Follow-up Cleanup Audit - 2026-07-15

Read-only audit of code cleanup, duplication, and componentization opportunities.
No code was changed; every item is a finding with a proposed fix, ranked and batched for a later decision.

## Method

Four specialized agents audited the codebase in parallel, one per region, each briefed with the maintainer's coding standards (comments explain *why* not *how*; DRY; prefer simplicity and maintainability):

- **Viewer frontend** - `viewer/js/*.js`, `app.css`, `index.html` (vendored `viewer/lib/` excluded).
- **CLI / server** - `src/Cli/Commands/*.cs`, `src/Cli/Services/*.cs`, `Program.cs`, `Types.cs`.
- **Sim / solver core** - `src/Sim/*.cs`, `src/Solver/*.cs`, `src/Extraction/MapExtractor.cs`.
- **Rig tooling** - `rig/*.mjs`, `rig/*.sh`, `rig/*.py`.

Scope was deliberately narrowed to code **added or changed since the 2026-07-10 audit** (`technical_debt.md`, whose 98 findings are all resolved).
That prior work is not re-litigated here.
The findings below concentrate in the areas that have grown most since: the textured-GLB pipeline, the multi-map server rewrite, the aim-reference feature, and the viewer's 3D/preview growth.

A subset of the highest-value claims were spot-verified by hand before writing this report; one was corrected (F19, `GrenadeRadius` is used by a test, not dead).

**Validation pass (2026-07-15, second sweep):** every finding in this report was re-verified against the source by hand.
All 23 hold as written - the damp-gate duplication (F1: `GrenadeTrajectory.cs:183` vs `:293`), the drifted Möller-Trumbore epsilons (F2: `1e-4f` in `TriangleRaycaster.cs:86` vs `1e-5f` in `TriangleCollider.cs:481`), the "mid"/"left+right" split (F3), the double `EYE_HEIGHT` (F4), the missing Escape handling (F5: zero `"Escape"` handlers existed in `viewer/js`), the light-theme contrast pair (F6: `#EA580C` on `#F7F9FA`), the triple reset-block (F12: `main.js:134/335/363`), the trig/predicate/constant duplications (F17, all four confirmed at the cited lines), the dead `RegionMin`/`RegionMax` (F21: no readers repo-wide), and the LineupApi magic numbers (F23) were each confirmed by grep or read.
No findings were removed; F19 remains the sole correction from the first pass.
Note: some F-numbered lines have since shifted or been resolved by the 2026-07-15 UX changes (F5's Escape gap is now partially closed - Escape cancels picking mode; the modals still need it).

## The recurring theme

Almost every finding is the same failure mode: a formula or block was copy-pasted, then one copy was edited and the other left stale.
Two instances are load-bearing and have already bitten or nearly bitten:

- The floor-impact damp gate (F1) was genuinely missing from one of its two copies for a period (commit `52f4d1b`), silently overshooting fast throws by ~120u.
- The ray-triangle intersection (F2) exists in two copies whose acceptance epsilon has already drifted 10x apart, with nobody able to say whether that is intentional.

The second theme is size: four files have grown into multiple responsibilities since the last split - `view3d.js` (893 lines), `ServeCommand.cs` (591), `LineupSolver.cs` (527), `TriangleCollider.cs` (492).

The third theme is that new UI (the map-picker and preview modals, the aim-reference badges) reintroduced accessibility gaps the prior audit had closed.

## P0 - correctness-adjacent (do first)

These are duplications that have caused or will cause silent bugs, plus two real accessibility regressions in new code.

### F1 - Floor-impact damp gate duplicated between the two integrators
`src/Sim/GrenadeTrajectory.cs:178-184` (voxel path) and `:284-294` (exact path).
The damp formula (`speed > gate && u > 0.5 && is-floor ? 1.5 - u : 1`) is written twice against two different normal representations.
This exact gate was already missing from the voxel copy once (commit `52f4d1b`), causing a ~120u overshoot on fast/steep throws that starved `VerifyExact`.
**Fix:** extract `static float FloorImpactDamp(preImpact, effectiveNormal, constants)`, called from both sites.
**Behavior risk: low** - arithmetic is equivalent but the expression evaluated changes; re-run golden replays after.

### F2 - Moller-Trumbore ray-triangle test duplicated with a drifted epsilon
`src/Sim/TriangleCollider.cs:452-491` (`HitTriangle`, accepts `t in (1e-5, 1]`) vs `src/Sim/TriangleRaycaster.cs:57-87` (`SegmentHitsTriangle`, accepts `t in (1e-4, 1 - 1e-4)`).
Same intersection math, but the `t` window has already diverged 10x at the near end with a different far-end convention, and nobody knows if that is intentional (sightline self-hit rejection) or accidental drift.
**Fix:** extract one `MollerTrumbore(origin, dir, a, b, c, tMin, tMax)` with the two epsilons passed in *explicitly* so the divergence becomes visible and deliberate.
**Do not** collapse the epsilons to one value without confirming - one window may be load-bearing.

### F3 - Click-label inconsistency: the same throw is "left+right" in one place and "mid" in another
`src/Cli/Services/LineupApi.cs:227` and `src/Cli/Commands/LineupsCommand.cs:171` say `"left+right"`; `src/Cli/Commands/ValidateCommand.cs:169,396` say `"mid"` for the identical 0.49-0.99 strength band.
The lineup API (what the viewer shows) and the validation report/plugin chat now describe the same click combo with two different words.
The same logic is also copy-pasted into the Python relays (`rig/relay-lineup.py`, `rig/relay-plineup.py`) - a cross-language third and fourth copy.
**Fix:** one `CliParsing.ClickName(float strength)` returning the canonical label (recommend `"left+right"`), used at all C# sites; make the Python relays internally consistent via `calibipc.py` (see F16).

### F4 - `EYE_HEIGHT` defined twice, with different type and a 0.06u mismatch
`viewer/js/state.js:85` exports `EYE_HEIGHT = 64` (a flat number, used by `panel.js:180` to convert a pasted getpos into a target); `viewer/js/view3d.js:483-484` declares its own module-local `EYE_HEIGHT = { Crouch: 46.04, ... }` plus `DEFAULT_EYE_HEIGHT = 64.06`.
Same identifier, different shape, and the standing value disagrees (64 vs 64.06).
Anyone editing "the eye-height constant" will silently miss the other copy.
**Fix:** move the per-type table + `DEFAULT_EYE_HEIGHT` into `state.js` as the single source of truth; have `view3d.js` import it and `panel.js` use `DEFAULT_EYE_HEIGHT` (64.06).

### F5 - The two dialogs added since the last audit have no focus management or Escape
`viewer/js/main.js:196-260` (`#intro`) and `:391-440` (`#preview-modal`); `index.html:133-157`.
`#intro` is shown with nothing focused into it and no Tab trap, so a keyboard/screen-reader user can tab straight into the blurred content behind it.
`#preview-modal` has no `role`/`aria-modal` at all, no Escape handler, and no focus restoration to the thumbnail that opened it.
Both are net-new since 2026-07-10 - exactly the regression class the prior audit closed.
**Fix:** one shared `openModal(el)/closeModal(el, returnFocusTo)` helper (focus in, Escape to close, Tab trap, focus restore); add `role="dialog" aria-modal="true"` to `#preview-modal`.

### F6 - `.ref.sky` aim-reference badge fails WCAG AA contrast in light theme
`viewer/app.css:495`.
The new (2026-07-14) sky-shot badge renders `--panel` text on a `--heat-none` (#EA580C) fill at 10px bold, measuring **3.37:1** in light theme - below the 4.5:1 minimum.
Dark theme is fine (~8:1), so it is easy to miss without checking both.
**Fix:** in light theme darken the text (e.g. `--ink` or a dedicated `--on-heat-none` tuned to 4.5:1), or use `--heat-none` as border+text only, matching how `.ref.reticle`/`.ref.edge` already do it.

## P1 - componentization / large-file splits

Mostly pure code motion; verify the build and tests are unchanged (bit-identical solves where claimed).

### F7 - Split `view3d.js` (893 lines) into focused modules
`viewer/js/view3d.js`.
`init3d()` alone is a ~400-line closure bundling scene/mesh bootstrap, marker/target/progress assets, a generic free-look+WASD camera controller (~280 lines with no dependency on the mesh or lineups), and the public `three` API.
**Fix (highest-value cut):** extract three modules -
- `viewer/js/flycam.js` - `createFlyCamera(camera, domElement) => { start, stop, resize, flyTo, topDown, requestRender }` (owns look/pan/dolly, `fly`, the `dirty` render-gate). This is the biggest single reduction.
- `viewer/js/textured-scene.js` - `ensureTexturedScene`/`resetEnsureTexturedScene` plus the material-classification logic currently inline in an anonymous `traverse` callback (`view3d.js:590-645`); naming it `classifyMaterial(mesh, vmat)` retires most of the 34-line explanatory comment above it (F13).
- `viewer/js/preview.js` - `renderPreview`/`capturePreview` (`view3d.js:660-768`) plus the `queuePreview`/thumbnail orchestration currently in `main.js:400-440`, so the preview feature lives in one place.

### F8 - Split `ServeCommand.cs` (591 lines) into three services
`src/Cli/Commands/ServeCommand.cs`.
One static class owns map discovery/versioning, Kestrel bootstrap, endpoint routing, static-file+ETag+cache-control policy, and gzip/brotli compression caching.
**Fix:**
- `Services/MapRegistry.cs` - `MapEntry` record, `LoadMaps`, brotli precompress, cache paths.
- `Services/StaticAssetServer.cs` - static serving, ETag/304, content-type table.
- `Services/HttpCompression.cs` - generic gzip/brotli byte helpers.
`ServeCommand.cs` then keeps only option parsing, host bootstrap, and route registration (~150-200 lines).
This split also naturally absorbs F20 (unused error helper) and F21 (duplicated 304 check).

### F9 - Split `TriangleCollider.cs` (492 lines) by concern
`src/Sim/TriangleCollider.cs`.
It mixes CSR bucket build, the sphere-approx query, the swept-hull SAT sweep, a second standalone SAT test (`BoxTriangleOverlap`), and ray-vs-mesh with two traversal strategies.
**Fix:** `partial class TriangleCollider` across `TriangleCollider.cs` (fields/CSR/`CellOf`/`Vertex`), `TriangleCollider.Sat.cs`, and `TriangleCollider.Raycast.cs`.
Pure code motion, zero behavior risk; enables F18.

### F10 - Split the origin-generation block out of `LineupSolver.cs` (527 lines)
`src/Solver/LineupSolver.cs`.
Lines 44-337 are the sweep/refine/rank algorithm; lines 339-526 are a self-contained origin-generation module (`OriginsFromNavAreas`, `NavGroundZ`, `PointInPolygon`, `SnapToGround`, `FindStandableOrigins`) with no dependency on the sweep except sharing the static class.
**Fix:** move it to `LineupSolver.Origins.cs` (`partial`), or a separate `OriginFinder` static class.
Zero behavior risk - same namespace, call sites unchanged.

### F11 - Extract `rig/glb-lib.mjs` for the duplicated NodeIO/Draco boilerplate
`rig/fix-prop-scale.mjs`, `rig/optimize-textured-glb.mjs`, `rig/revert-node-scale.mjs`, `rig/roundtrip-test.mjs`.
The same ~11-line block (three imports, `Promise.all([createDecoderModule, createEncoderModule])`, the `NodeIO().registerExtensions(...).registerDependencies(...)` build, `io.read`) is copy-pasted into all four; `roundtrip-test.mjs` is *entirely* this boilerplate.
**Fix:** `rig/glb-lib.mjs` exporting a memoized `getIO()` + `readGlb`/`writeGlb`, plus (opportunistically) the reusable `median`/`meshExtent`/`vmatName`/`groupNodesByMesh` helpers from `fix-prop-scale.mjs:117-127` and an argv-validation helper.
Note F17 removes two of the four consumers.

## New findings - 2026-07-15 persona review (game-dev, CS2 player, esports coach, FPS-gamer UX)

Code-anchored issues surfaced by the four persona agents and verified against source.
Items marked *(fixed 2026-07-15)* were resolved in the same-day UX pass; the rest are open.

### Correctness / stale-content bugs
- **F24 - `Describe()` recommends a dead input method.** `src/Cli/Services/CliParsing.cs:124-141` says "jumpthrow bind" for every jump-type lineup, but Valve disabled multi-input binds on official servers (Aug 2024, `cl_allow_multi_input_binds 0`); players now time it manually (subtick gives ~50ms forgiveness) or use scroll-jump. The physics is unaffected - the guidance text is what's stale. High user-trust impact.
- **F25 - static 3D hint text was wrong after the first target** *(fixed 2026-07-15: hint is now derived from state)*. `main.js` advertised "click terrain = set target" permanently, but once a target existed a click actually launched a solve.
- **F26 - intro copy contradicted the armed-picking flow** *(fixed 2026-07-15: a plain click now sets the first target directly, making "click anywhere on it" true)*.
- **F27 - filter labels/descriptions render once in the intro, then are discarded.** `main.js` `mountIntroFilters()` builds label+description wrappers only inside the intro modal; `closeIntro()` re-parents the bare selects and the explanations are never seen again (returning users never see them at all). Open.
- **F28 - 3D camera pan/dolly were unreachable on touch** *(fixed 2026-07-15: two-finger drag pans, pinch dollies, `touch-action: none` added)*. Touch can produce neither a secondary button nor wheel events; the pan/dolly branches were desktop-only.
- **F29 - zero-result message claimed an exhaustive sweep for single-origin probes** *(fixed 2026-07-15: message now branches on whether an origin was supplied)*.

### UX debt (open, ranked)
- **F30 - first preview click can trigger an 18-46MB GLB download with no size/progress framing** (`main.js` preview path lazily loads the textured GLB; only feedback is a pulsing "rendering preview…"). Needs a byte-progress readout and one-time size expectation.
- **F31 - sky-aim filter (≤65°) is silently active on the very first solve**; the only cue is a small count on a collapsed accordion. Surface "N hidden by filters" in the results status line.
- **F32 - no visible reset-view control on the 2D map**; recovery from a lost pan/zoom is an undocumented double-click.
- **F33 - touch targets don't scale on mobile**: buttons stay ~20px tall under the 640px breakpoint and `PICK_RADIUS_PX = 12` is identical for mouse and touch (should be ~20-24px for `pointerType === "touch"`).
- **F34 - aim-reference badges (SKY/reticle/flat/edge) are explained only via `title` tooltips**, which never fire on touch; fold them into the existing legend chip.
- **F35 - no side-by-side lineup comparison**; selection is a single index, so comparing two candidates means toggling back and forth.

### Product feature gaps (persona consensus, for roadmap - not cleanup)
Ranked by how often the personas independently converged on them:
1. **Shareable permalink** encoding map + target + selected lineup (all four personas; the CS2 player called it the single change that turns a solo tool into a team tool).
2. **Persistent, nameable favorites** ("my nades") - favorites are session-only object flags today; coach wants named *sets* (executes) on top.
3. **Throw-to-bloom timing** on the card - flight time ships, bloom delay data is already captured per BACKLOG.md but unanalyzed; needed for sequencing utility.
4. **Expose multi-target solving in the web UI** - the CLI already seals multi-sightline chokes (semicolon targets); the web UI only sends one target.
5. **Throw-spot exposure scoring** on the heatmap ("safe smoke" filter) - the heatmap is the tool's differentiator; ranking origins by cover would make it a coaching superpower.
6. **One-way smoke search** - the occlusion model is symmetric today; per-eye sampling already exists, so asymmetric (one-way) detection is feasible.
7. Position-reference half of the aim-reference system (corner-wedge vs open ground), demo-position import, per-player assignment/print view for coaches.

## Feature plan: shareable permalinks (scheduled AFTER all findings above are done)

The one change all four persona reviewers converged on.
Goal: any solved lineup is a URL you can paste to a teammate, who lands on the same map, same target, same selected lineup - no accounts, no server-side storage.

### URL contract

Extend the existing `?map=` param (the only URL state today) with two optional params:

- `t=x,y,z` - the target in world units (the server-resolved target, which carries the snapped ground Z, so the link never re-fights the 2D Z-ambiguity).
- `l=type:strength:fx:fy:fz:pitch:yaw` - the selected lineup identified by its *physical* parameters (feet position, throw type, click strength, angles), NOT by its index.
  Result indices are unstable across data redeploys and filter changes; the physics parameters are the lineup's identity - the same values `/api/trajectory` already accepts.

Example: `smoke.npc-server.top/?map=de_mirage&t=-50,-1122,12&l=JumpThrow:1:-1024:-908:-40:-32.5:145.2`

### Load behavior

1. `map` loads as today; `t` sets the target directly (skips the intro).
2. With `l` present: run a **single-origin probe** at the lineup's feet (cheap, and the query cache means a shared link is usually a cache hit) and auto-select the result whose feet/type/strength/pitch/yaw match within epsilon (feet 1u, angles 0.1 deg). If nothing matches (map data changed since the link was minted), fall back to a full `Search map` and show "the linked lineup no longer exists on current map data - showing all lineups for its target".
3. With only `t`: auto-run the full sweep, exactly as if the user had clicked Search map - a progress stream on landing is the feature, not a cost.

### Write behavior

- `history.replaceState` keeps the address bar current as the user acts (target set/moved -> update `t`; lineup selected -> update `l`; cleared -> params dropped). Copying the address bar is always a valid share.
- A **Share button** on the selected-lineup card: `navigator.share` where available (mobile), clipboard copy with a "link copied" toast elsewhere.

### Non-goals (v1)

- No server-side link storage, no shortener, no accounts - the URL is self-contained.
- No multi-lineup sets (that is the coach's "executes" feature, a separate later item).
- Filters are deliberately NOT encoded: the link recipient sees the linked lineup even if their own default filters would have hidden it (auto-selection bypasses filters and surfaces a note instead).

### Verification

Round-trip E2E: solve, select, copy link, open in a fresh profile (no localStorage), assert same map/target/selected lineup; plus a stale-link test against a mutated cache key.
